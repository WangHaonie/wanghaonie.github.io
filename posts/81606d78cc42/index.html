首先我以为是网络问题，于是我 Ping 了一下 github.com，结果如下

果不其然，这肯定是超时了的，于是到ipip.net测试了 github.com 全球的服务器，
目的是为了找出能 Ping 通的 CDN 节点，最终找到了这几个 IP 地址：

  • 13.114.40.48 (日本)
  • 140.82.114.4 (美国)
  • 140.82.121.4 (美国)

又分别在自己的电脑上 Ping 了一下

可以看出，这 3 个 CDN 节点是可以被 Ping 通的，其中日本的延迟最低，要更改修改系统 Hosts

修改 Hosts 文件的方法：

1.运行 notepad C:\Windows\system32\drivers\etc\hosts 打开记事本来修改。

2.或者打开 cmd 运行 echo 13.114.40.48 github.com >>C:\Windows\system32\drivers\etc\hosts 直接快速修改

完事之后，重新执行 hexo clean &&hexo g &&hexo d，如下

最后用了一段时间，发现又是同样的报错，根据 教程 来看，那么肯定是 SSH Key 失效了，我也想过为什么会无缘无故的失效，终究没有头绪，不管了，我直接又重新生成了 id_rsa.pub，然后到 SSH and GPG keys 删除又重新创建了一个新的 SSH Key，再去执行，又成功了。

总结一下注意事项

  1. 一定要关闭电脑里面的全局代理，特别是 Clash for Windows 的 TUN Mode (网卡模式)，必须关闭，系统代理都可以不用关，不然会影响 hexo d 的执行，其他命令 hexo init、npm、git 等不受影响，打开了还会加快下载速度。

[2022-08-27] 不一定要关，具体看节点接不接受这个协议，我近期换了另一个节点就可以代理 ssh 了，速度飞起，也不用改 hosts 了。

  1. 对于 hexo 的一系列命令，推荐在 Git Bash 里面执行，用 cmd 执行会报上面那个错误，可能是在 git 里面设置了 --global 参数的缘故吧，然而 hexo 又可能依赖这些参数，用 cmd 获取不到这些参数，自然就会报错了。(未经证实，仅供参考)
  2. 每次修改完配置文件或发完文章后，必须执行 hexo clean &&hexo g &&hexo d，网上说可以直接用 hexo d，结果我的 index.html 和其他所有的 HTML 文件都不见了，很奇怪。
